# プロダクション防衛・監査強化ワークフロー (Production Safety Check)

AIによる20回以上のコード監査をすり抜けた原因は「見た目のUI監査」と「TypeScriptの型補完機能の無効化（Type Blindness）」に依存していたためです。
今後、同種の「低レベルな不具合」を撲滅するため、すべての機能実装・改修後に以下の防衛ラインを**必ず自律的に完全実行**します。

## 1. 非同期処理のロールバック保証（Async State Rollback）
- **対象**: `setMessages` などの楽観的UI（Optimistic UI）更新と、裏側での非同期APIアクセスが混在する処理。
- **監査事項**: APIエラー時（例: `success: false`）にUIが確実に元の状態に復元（Rollback）されるかを確認する。エラーハンドリングが握りつぶされていないかを検査。
- **実例**: メッセージ削除処理において、手前でUIから消したにも関わらずAPIが失敗した場合、削除されたメッセージが亡霊のように復活する現象の防御。

## 2. Type Blindness（型盲の自動検知と排除）
- **対象**: データベースフェッチ結果などにおける `as unknown as`, `as any` による強制キャスト。
- **監査事項**: このキャストはTypeScriptのコンパイラを黙らせるため、「オブジェクトだと思っていたら配列だった」などの致命的バグ（例：あなた、がゲストに変わるバグ）を100%見逃します。
- **アクション**: キャストを発見した場合、`Array.isArray(data) ? ...` などの厳密なランタイム型チェック（Zodパターンの導入）を必ず実装するか、Supabase CLIの公式Typeを使用する。

## 3. 状態遷移の競合チェック（Race Condition Check）
- **対象**: `useMemo` によるリアクティブ依存と、即時状態書き換えが競合する領域。
- **監査事項**: 新規追加中（is_optimistic）に削除操作や編集操作が発行された場合、システムがクラッシュしないか、操作パネルが正しく隠蔽されているかを確認する。

## 4. TypeScript Strict Build
- 最後に必ず `npm run build` を実行し、暗黙の `any` や依存配列の抜け漏れ（useEffect / useMemo）が存在しないことをシステムレベルで担保してから完了報告を行う。
