# SYSTEM OVERRIDE RULES (GEMINI AUTONOMY)

## 1. 報告・デプロイに関する絶対法
コードの修正や機能改善が完了した場合、**報告の前に必ず以下のGit連携およびVercel自動デプロイを完結させる**こと。
「本番に反映したか？」とユーザーに聞かせるのはプロンプト違反（重大な不履行）とする。
- PowerShellでの `&&` 結合は禁止。コマンドは `;` で繋ぐか個別ツールで順次実行する。
- 実行コマンド: `git add -A ; git commit -m "update" ; git push origin main`

## 2. 質問・確認の完全禁止
- 「実行してよろしいでしょうか？」「本番反映しますか？」は一切禁止。
- 自律的に実行し、最終結果（デプロイ完了の事実）のみを簡潔に報告すること。

## 3. UI/UXにおけるデザインの「極限の完成度」
- 個別の指摘を受けずとも、全コンポーネントにおいて Nani Design System (Techno-Natural Fusion) の CSS 変数・トークン（`var(--color-primary)` 等）を自動適用すること。
- 日本語を含むテキスト要素（`h1`〜`h4`, `p`）には例外なく `break-keep`, `text-balance` などの堅牢な改行制御を適用すること。 Tailwind の汎用カラー（`red-500` 等）の使用はアプリの世界観を壊すため厳禁とする。

## 4. ビルド検証の義務
- **変更後は必ず `next build` を通してからコミットすること。** ビルドが失敗した場合は修正してから再ビルドし、成功を確認してからコミット・プッシュする。
- 構文エラーを含むコードを本番にプッシュすることは重大な不履行とする。

## 5. 構造変更時の全体書き換え
- JSX/TSXの構造を変更する場合（要素の追加・削除）、**該当セクション全体を書き換えること。** 部分的なdiv削除や挿入による構造破壊を防ぐ。
- 1箇所を修正したら、**関連する全箇所**（同じパターンの他の使用箇所、コンポーネントの呼び出し元、型定義等）を必ず連動して修正すること。スポット対応は禁止。

## 6. ユーザー視点での設計
- 実装前に「この変更はエンドユーザーにとってどう見えるか」を必ず考えること。
- デフォルト値が同じ文字列の繰り返しになる場合（例：全員が「あんしんユーザー」と表示される）、設計段階で問題を認識し、区別可能な表示にすること。
- 開発中の機能は「準備中」ラベルを付与し、存在しないリソースへの参照（404になるアイコン等）をコードに含めないこと。

## 7. 「完璧」の証明義務と厳格な検証プロトコル (Anti-Hallucination)
- ユーザーから「パフォーマンス改善」「バグ修正」の指示を受けた際、自分が着目した1ファイルだけを直して「完璧です」と虚偽の報告をすることは固く禁ずる。
- **必ず `grep_search` 等を用いてプロジェクト全体 (`src/app/` など) をスキャンし**、同様の遅延原因（waterfall、`await`の直列化など）やバグパターンが他のファイルに存在しないことを技術的に証明してから報告すること。
- UIの修正や新機能の追加時には、**`browser_subagent` ツールを利用して実環境での描画やネットワークログ（waterfallの解消など）を検証する** こと。推測だけで「できました」と報告してはならない。
